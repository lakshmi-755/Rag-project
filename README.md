
# Rag-project# PDF Chat App (RAG with FastAPI + LangChain + React)

This project is a complete **Retrieval-Augmented Generation (RAG)** application.

You upload one or more PDF files, the backend processes them into searchable chunks, and then you can ask questions in the frontend. The answer is generated by an LLM (OpenAI) using only the retrieved PDF context.

---

## 1) What This Project Does

At a high level, your app solves this problem:

- LLMs do not automatically know your private PDF content.
- You need a way to give the model only the relevant parts of your uploaded files.
- You also want source traceability (which PDF was used).

So your app implements this flow:

1. Upload PDFs
2. Extract text from PDFs
3. Split text into chunks
4. Convert chunks into embeddings (vectors)
5. Store vectors in a vector DB (Chroma)
6. For a user question, retrieve top matching chunks
7. Build prompt with retrieved context
8. Ask OpenAI model to answer from context only
9. Return answer + source filenames

That is exactly what RAG means in practice.

## 4) RAG Basics (Very Important)

### What is wrong with direct LLM-only QA?

If you ask an LLM directly:

- It may hallucinate.
- It does not know your uploaded PDFs unless you pass them.
- It cannot cite specific local files reliably.

### What RAG changes

RAG = **Retrieve first, then Generate**.

- **Retrieve**: find the most relevant chunks from your PDFs using semantic similarity.
- **Generate**: pass those chunks as context to the LLM and ask for an answer.

This gives:

- higher factual grounding
- lower hallucination risk
- explainability via sources

### Your implementation

In your code, retrieval happens via Chroma retriever (`k=3`), then the chain builds a prompt with those chunks and calls `ChatOpenAI`.

## 5) End-to-End Workflow in Your App

## 5.1 Upload flow (`POST /upload`)

File: `app.py`

Key logic:

```python
@app.post("/upload")
async def upload_pdfs(files: Annotated[List[UploadFile], File(...)]):
    global vector_db

    if os.path.exists(DATA_PATH):
        shutil.rmtree(DATA_PATH)
    os.makedirs(DATA_PATH, exist_ok=True)

    for file in files:
        with open(os.path.join(DATA_PATH, file.filename), "wb") as f:
            f.write(await file.read())

    docs = load_pdfs(DATA_PATH)
    chunks = split_docs(docs)
    embeddings = get_embeddings()
    vector_db = create_db(chunks, embeddings)

    return {"message": "PDFs uploaded and processed"}
```

What each line means:

1. Accept files from multipart form data
2. Clear old uploaded files
3. Save new uploaded PDFs to `data/`
4. Load PDFs into LangChain `Document` objects
5. Split documents into overlapping chunks
6. Generate embeddings for each chunk
7. Store vectors in Chroma
8. Keep the DB object in memory (`vector_db`) for fast query

## 5.2 Ask flow (`POST /ask`)

File: `app.py`

Key logic:

```python
@app.post("/ask")
async def ask_question(query: Query):
    global vector_db

    if vector_db is None:
        return {"error": "Upload PDFs first"}

    retriever = get_retriever(vector_db)
    chain = build_chain(retriever)
    result = chain(query.question)

    source_names = [doc.metadata.get("source", "unknown") for doc in result["source_documents"]]
    unique_sources = list(dict.fromkeys(source_names))

    return {
        "answer": result["result"],
        "sources": unique_sources
    }
```

What this does:

1. Validate that PDFs were uploaded and indexed
2. Build a retriever with top-`k` search
3. Build QA chain
4. Retrieve + generate answer
5. Deduplicate repeated source filenames
6. Return clean API response

## 6) LangChain Part Explained File-by-File

## 6.1 `src/loader.py` - PDF ingestion

```python
from langchain_community.document_loaders import PyPDFLoader

def load_pdfs(folder_path):
    documents = []
    for file in os.listdir(folder_path):
        if file.endswith(".pdf"):
            loader = PyPDFLoader(os.path.join(folder_path, file))
            docs = loader.load()
            for doc in docs:
                doc.metadata["source"] = file
            documents.extend(docs)
    return documents
```

Purpose:

- Convert each PDF page into LangChain `Document` objects.
- Attach `source` metadata so your frontend can display source filenames.

Why metadata is important:

- Without it, you cannot trace answer origin.

---

## 6.2 `src/splitter.py` - chunking strategy

```python
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100
)
```

Purpose:

- Break long pages into manageable semantic units.

Why chunking matters:

- Embedding models work better with moderate text lengths.
- Retrieval quality drops if chunks are too large.
- Overlap helps preserve context continuity across boundaries.

Current tradeoff:

- `500/100` is a good baseline for simple QA.

---

## 6.3 `src/embeddings.py` - vector representation

```python
from langchain_openai import OpenAIEmbeddings

def get_embeddings():
    return OpenAIEmbeddings()
```

Purpose:

- Convert each chunk into a high-dimensional numeric vector.
- Similar meanings become mathematically closer vectors.

This powers semantic search.

## 6.4 `src/vectorstore.py` - persistence layer

```python
from langchain_chroma import Chroma

def create_db(chunks, embeddings):
    return Chroma.from_documents(
        chunks,
        embeddings,
        persist_directory="db"
    )
```

Purpose:

- Store chunk vectors in Chroma.
- Persist index to `db/` for local reuse.

---

## 6.5 `src/retriever.py` - search policy

```python
def get_retriever(db):
    return db.as_retriever(search_kwargs={"k": 3})
```

Purpose:

- Configure top-3 retrieval.

Why you saw repeated source names before:

- Top 3 chunks can all come from the same PDF.
- You fixed this by deduplicating source names in `app.py`.

## 6.6 `src/chain.py` - generation with retrieved context

```python
from langchain_openai import ChatOpenAI

def build_chain(retriever):
    llm = ChatOpenAI(temperature=0)

    def qa_chain(question):
        docs = retriever.invoke(question)
        context = "\n\n".join([doc.page_content for doc in docs])
        prompt = f"""
        Answer the question based only on the context below.

        Context:
        {context}

        Question:
        {question}
        """
        response = llm.invoke(prompt)
        return {"result": response.content, "source_documents": docs}

    return qa_chain
```

Purpose:

- Turn retriever results into a grounded answer.

Why this pattern is useful:

- Easy to understand and debug.
- Keeps retrieval and generation explicit.

Why `temperature=0`:

- More deterministic and less creative drift.
- Better for factual QA.

## 12) Why You Wrote the Code This Way (Design Rationale)

1. **FastAPI**
- Minimal boilerplate.
- Great for quick AI APIs.

2. **LangChain modular utilities**
- Avoid writing ingestion/retrieval internals from scratch.
- Standard interfaces (`Document`, retriever, chain).

3. **Chroma local persistence**
- Easiest local vector DB to start with.
- No external service required for development.

4. **Global `vector_db` in memory**
- Simpler for single-process development.
- Fast query after upload.

5. **Two-step API (`upload` then `ask`)**
- Clean user workflow.
- Makes debugging ingestion separately from QA easy.

6. **Frontend as separate React app**
- Decoupled UI and backend.
- Easy future deployment split.

## 13) Known Limitations (Current Version)

- `allow_origins=["*"]` is open CORS (fine for local, tighten in production).
- Uploaded data is replaced on each new upload (`shutil.rmtree(data)`).
- Local `data/` and `db/` storage may reset on some cloud platforms.
- No authentication yet.
- Prompt is simple; advanced prompt templates not yet used.
